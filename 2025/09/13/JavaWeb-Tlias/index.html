3. 形参注解

3.1 @RequestParam

用途：@RequestParam 用于从请求的 URL 查询参数中提取参数。它通常用于处理 GET 请求时的参数传递。(问号参数)

常见用法：常用于查询操作，或者在 URL 中带有多个查询参数的场景。

示例：

请求：GET /api/users?age=25&name=John

3.2 @PathVariable

用途：@PathVariable 用于从 URL 路径中提取参数。它通常用于动态路径，常见于 RESTful 风格的 URL 中。(路径参数)

常见用法：常用于单个条件查询、按id删除等操作，获取 URL 路径中的动态部分（如资源 ID）。

示例：

请求：GET /api/users/123

3.3 @RequestBody

用途：@RequestBody 用于从请求的正文中提取数据，通常用于处理 POST 或 PUT 请求中的 JSON 或其他格式的请求体。（添加、修改操作）

常见用法：通常用于提交表单数据或提交较复杂的数据结构。

5.事务

5.1 简介

为了防止数据库更新的不一致性，比如一个请求需要同时修改员工信息和员工工作经历信息，需要使用事务，使对数据库的两个操作要么都成功，要么都失败。

用法：在业务方法上添加注解@Transactional

属性1：rollbackFor

控制事务生效时机：该注解默认只出现RuntimeException(运行时异常)才会回滚事务。要想所有异常均会回滚，需要加上属性rollbackFor，即@Transactional(rollbackFor = Exception.class)

属性2：propagation （默认值是REQUIRED）

控制事务传播性，加入现在有两个事务方法A、B，且A方法调用B方法。默认情况下B会加入到A事务中，即A、B两个方法涉及到的对数据库的操作要么都成功，要么都失败。要想将B事务从A事务中隔离出来，需要用到属性propagation，即@Transactional(propagation = Propagation.REQUIRES_NEW)

5.2 事务的四大特性

原子性：事务里的操作要么都成功，要么都失败

一致性：数据库的数据要么都变化，要么都回滚

隔离性：多个用户并发的访问数据库时，一个用户的事务不能被其他用户的事务干扰，多个并发的事务之间要相互隔离。一个事务的成功或者失败对于其他的事务没有影响。

持久性：事务一旦被提交或回滚，数据库的改变将是永久性的。
